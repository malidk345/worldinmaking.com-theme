## 2024-05-18 - Date Instantiations Inside Render and Sort Loops
**Learning:** React components (like the `PostsView` memoized context) and frontend utilities (like `usePosts`) often loop over items and sort them by date. Calculating `.getTime()` or `unix()` from strings using new `Date()` or `dayjs()` inside the comparison function of `.sort((a,b) => ...)` results in `$O(N \log N)$` redundant date instantiations per item. In large data arrays, this degrades client-side sorting and blocking time.
**Action:** Always pre-calculate dates inside an initial `.map()` traversal ($O(N)$) before applying `.sort()`, utilizing the Schwartzian transform: `.map(item => ({ item, time: dayjs(item.timestamp).unix() })).sort(...).map(m => m.item)`. I applied this optimization successfully in `PostsView`, `usePosts`, and `NotificationCenter`.

## 2024-07-24 - Avoiding O(C*P) iterations and Date parsing during render
**Learning:** Performing `Array.filter` followed by `Array.reduce` inside a render loop for every category/channel (C) against all entities (P) creates an O(C*P) bottleneck. Furthermore, calling `new Date(string).getTime()` continuously inside this nested loop is highly expensive. Because ISO 8601 strings (e.g. `created_at` from Supabase) are strictly formatted from largest time unit to smallest (YYYY-MM-DDTHH:mm:ss), they can be accurately compared using standard lexicographical string comparison.
**Action:** When finding the latest or earliest items associated with categories from a flat array, iterate through the flat array once O(P) inside a `useMemo` block to build a `Map` of the extreme values per category. Additionally, always compare ISO 8601 datetime strings directly rather than instantiating `Date` objects unless specific date math/formatting is immediately required.
