## 2024-05-18 - Date Instantiations Inside Render and Sort Loops
**Learning:** React components (like the `PostsView` memoized context) and frontend utilities (like `usePosts`) often loop over items and sort them by date. Calculating `.getTime()` or `unix()` from strings using new `Date()` or `dayjs()` inside the comparison function of `.sort((a,b) => ...)` results in `$O(N \log N)$` redundant date instantiations per item. In large data arrays, this degrades client-side sorting and blocking time.
**Action:** Always pre-calculate dates inside an initial `.map()` traversal ($O(N)$) before applying `.sort()`, utilizing the Schwartzian transform: `.map(item => ({ item, time: dayjs(item.timestamp).unix() })).sort(...).map(m => m.item)`. I applied this optimization successfully in `PostsView`, `usePosts`, and `NotificationCenter`.

## 2024-05-18 - Avoid O(N*M) Date parsing in loops
**Learning:** React component iteration that calculates derivatives from large arrays (`posts`) inside `.map` calls across smaller arrays (`channels`) creates a hidden O(C*P) complexity. Furthermore, Supabase returns timestamps as ISO 8601 formatted strings which are lexicographically sortable. Instantiating `new Date(post.created_at).getTime()` for comparison inside high-frequency render functions or reduce callbacks drastically decreases rendering performance due to expensive garbage collection and parsing.
**Action:** When deriving metrics grouped by an identifier (e.g., latest post per channel), always compute them in a single O(P) pass via a `useMemo` block using `Map` or `Record`. Exploit ISO 8601 lexicographical properties with direct string comparison (`post.created_at > currentLatest`) to completely bypass `Date` instantiation.
